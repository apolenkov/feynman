# RTK Playbook для Codex (feynman)

RTK = Run-Task Kind. Короткая схема, чтобы держать скорость и не перегревать контекст.

## 1) Классификация задачи

| Класс | Признаки | Цель | Рекомендуемый режим |
|---|---|---|---|
| **Tiny** | Документация, правки 1–3 файлов, без архитектурных решений | Быстро закрыть | `low` reasoning, без запуска подпроцессов |
| **Normal** | Несколько файлов, есть зависимые эффекты, нужны проверки в 1–2 местах | Стабильно довести флоу | `medium`/`high` reasoning, 1–2 параллельные ветки |
| **Complex** | Новая фича, refactor нескольких модулей, риск регрессий/edge-case | Полный вывод с рисками | `high` reasoning, параллельные параллельные агентов |

## 2) Как запускать Codex/агентов по задаче

- **Tiny**: локально, без долгих спонов, сразу edit/patch + короткая проверка.
- **Normal**: использовать `parallel` для независимых проверок/чтений/подготовок.
- **Complex**: спавнить агентов по типам задач:
  - исследовательский контур (контур документа/архитектуры),
  - исполнительный контур (код/тесты),
  - ревью-контур (проверка рисков и регрессий).

## 3) Модельный профиль (что реально возможно в текущей конфигурации)

- В текущем runtime Codex не умеет автоматически менять модель на уровне одного запроса по сложности без отдельной маршрутизации;
- Поэтому для "легко/тяжело" используем:
  - **ручной уровень анализа** (Tiny/Normal/Complex),
  - **разный объём работы параллельно**,
  - **разный уровень `reasoning_effort`** в специализированных подпроцессах.
- Идея для будущего: добавить явный рутинг (mini/spark/strong) в wrapper-команду или GitHub-оркестратор.

## 4) Как экономить токены при диаграммах (сразу включается в prompt-контекст)

- Сначала реши, нужна ли диаграмма по этой структуре:
  - Да: последовательность, дерево, сравнение, статусная сводка или приоритет.
  - Нет: одна-две фактические фразы без структуры.
- В каждом ответе держи диаграмму минимальной:
  - максимум 3 ветки/уровня в одном блоке,
  - короткие метки (до ~6 слов),
  - не дублируй текстовую таблицу + диаграмму для одного и того же блока.
- Для короткого статуса (до 3 элементов) лучше 1-2 строчные метки и
  маленький frame; длинную наррацию не надо.
- Если блок не выигрывает по ясности от диаграммы — лучше plain-text.

## 5) Что запускать в первую очередь

- `feynman examples` — запускать как пример шаблона проверки UX/диаграмм.
- По каждому Normal/Complex: отдельный статус-обзор + перечень рисков + проверочный список.
- После флага/перехода: краткий manual-check + CI/тесты только по затронутым зонам.

## 5) Практическое правило для вашего процесса

1. `Tiny`: не больше 1–2 инструмента и без спавна.
2. `Normal`: спавнить 1–2 простых агентов параллельно.
3. `Complex`: обязательно 2+ агентов параллельно + один ридер/ревью-контур.

## 6) `feynman` slash-команды (alias policy)

- `/feynman on` и `/feynman start` — одно и то же: включить инъекцию.
- `/feynman off` и `/feynman stop` — одно и то же: отключить инъекцию.
- `/feynman lite` — включить в lite-режиме.
- `/feynman full` — включить в full-режиме.
- `/feynman ultra` — включить в ultra-режиме.
