# Q-2026-05-09-01 — XML Tag Compatibility Across feynman Hosts

**Question.** Do Codex (CLI hooks), Cursor (`.mdc`), Cline (`.clinerules/`), and
Windsurf (`.windsurf/rules/`) handle XML-tagged sections inside injected rule
content the same way Claude Code does? If we migrate `feynman-activate.md` from
`<!-- lite -->...<!-- /lite -->` Markdown comments + headers to
`<intensity name="lite">...</intensity>` and `<structure_triggers>...`, do we
risk degrading non-Claude hosts?

## Evidence by host

**Codex CLI.** The `UserPromptSubmit` hook returns
`hookSpecificOutput.additionalContext`; the core pipeline preserves it as a
raw string (newlines intact) and maps each entry to a
`DeveloperInstructions::new(additional_context)` payload sent to the model
(GitHub `openai/codex#16486`). Only the TUI render layer collapses display —
the model sees the full text. OpenAI's own prompt guidance (developers.openai.com,
GPT-5 / GPT-5.1 / GPT-4.1 cookbook guides, 2025 model spec) recommends XML
delimiters and notes "improved adherence to information in XML with recent
models." `PreToolUse.additionalContext` is not yet supported (issue #19385),
but `UserPromptSubmit` — which is what feynman uses — is fully supported.

**Cursor.** Multiple 2025–2026 community guides and the Cursor docs converge:
"if you want rules to actually be followed, use pseudo-XML tags." The
recommended schema literally uses `<rule>`, `<requirements>`, `<example>`,
`<correct-example>` tags inside `.mdc` body. `.mdc` body is plain Markdown
after YAML frontmatter — XML passes through untouched.

**Cline.** Official `prompts/.clinerules/writing-effective-clinerules.md`
notes "benefits to writing agent rules in pseudo-XML, in part due to
increased model adherence to instructions within XML structures." The only
gotcha is filename: Cline rejects a `.xml` extension on rule files (issue
#5156) — but XML *content* inside an `.md` file is fully supported.

**Windsurf.** Cascade rules docs explicitly recommend XML for grouping:
"XML tags can be an effective way to communicate and group similar rules
together... `<coding_guidelines>...</coding_guidelines>`." Hard limit
6 000 chars per file / 12 000 chars total — feynman's intensity sections
fit comfortably.

**Cross-cutting model evidence.** OpenAI Cookbook (GPT-5 prompting guide,
GPT-4.1 prompting guide), OpenAI 2025-12-18 Model Spec, and the OpenAI
Developer Community thread "Use XML tags to structure my prompts" all
treat XML delimiters as a first-class structuring technique on GPT-4o /
GPT-4.1 / GPT-5 / GPT-5.1. Quote from the OpenAI prompt guidance: the
model is "trained to robustly understand structure in a variety of
formats" and XML "performs well." Anthropic's training advantage on XML
exists, but it is not exclusive — XML-affinity is now industry-wide.

## Recommendation

**Ship XML migration globally** — single rule file, no per-host fork.

```
[feynman-activate.md] ──┬──▶ Claude Code  (XML-native)
                        ├──▶ Codex CLI    (raw string → GPT XML-safe)
                        ├──▶ Cursor .mdc  (XML is best-practice)
                        ├──▶ Cline        (XML recommended)
                        └──▶ Windsurf     (XML documented pattern)
```

Rationale: every host preserves XML tags verbatim through its injection
mechanism, and every host's underlying model family has documented support
for XML-delimited instructions. Forking the rule file would double
maintenance burden with no measurable upside. The only Cline-specific
constraint (no `.xml` extension) is irrelevant — feynman ships `.md`.

**Caveats to verify in a 1-day spike post-migration:**
1. Windsurf 6k/12k char ceiling — measure rendered file size after XML
   wrapping; trim `ultra` intensity if needed.
2. `additionalContext` 10 000 char cap on Claude Code — XML wrapping adds
   ~5 % overhead; current `full` intensity at ~3 KB has headroom.
3. Codex TUI display still collapses multiline — operator-visible only,
   does not affect model behavior. Document in README.

## Decision impact

- **Ship:** single XML-tagged `rules/feynman-activate.md` for all five
  hosts. No per-host file split.
- **Schema:** `<intensity name="lite|full|ultra">…</intensity>` replaces
  `<!-- lite -->…<!-- /lite -->` markers. `<structure_triggers>`,
  `<rule>`, `<example>` for inner grouping.
- **Parser change:** intensity extractor in `feynman-activate.js` switches
  regex from HTML-comment markers to XML element matching. Schema in
  `state.json` (`intensity` field) is unchanged — backward compatible.
- **CLAUDE.md update needed:** "Формат файла правил" section currently
  documents `<!-- lite -->` markers; rewrite to XML element form.
- **Spike before merge:** char-count audit against Windsurf 6k/12k cap and
  Claude `additionalContext` 10k cap.
- **No risk identified for Codex / Cursor / Cline / Windsurf** under the
  evidence above; ship is GO.
